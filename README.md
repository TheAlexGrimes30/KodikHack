# Kodik Project — Система калиброванных ставок

## Описание проекта

Система калиброванных ставок — интеллектуальная платформа поддержки принятия решений при запуске новых продуктов, фич или бизнес-инициатив в условиях неопределённости.

Основная идея проекта — заменить одно большое рискованное решение серией небольших проверяемых ставок. Каждая ставка проходит анализ, выполняется на практике и после получения фактических результатов влияет на доверие к следующим решениям.

Архитектура проекта основана на пяти основных компонентах:

1. **Намерение (Intent)** — описание идеи и извлечение скрытых допущений.
2. **Среда (Environment)** — моделирование внешних факторов и сценариев.
3. **Деструктор (Attack)** — поиск наиболее вероятных причин провала.
4. **Калибровка (Calibration)** — сравнение прогноза с реальным результатом.
5. **Порог (Threshold)** — принятие решения о следующем шаге.

# Мультиагентная система

В проекте реализована **иерархическая мультиагентная система (Hierarchical Multi-Agent System)** на основе **LangGraph**.

Каждый агент отвечает только за одну предметную область.

Все агенты взаимодействуют через единое состояние (`AgentState`).

Архитектура анализа:

```
                 User
                   │
                   ▼
           Intention Agent
                   │
                   ▼
          Environment Agent
                   │
                   ▼
           Destructor Agent
                   │
                   ▼
            Threshold Agent
                   │
                   ▼
                  END
```

После получения реальных результатов проверки гипотезы запускается отдельный граф:

```
START
   │
   ▼
Calibration Engine
   │
   ▼
END
```

---

# Общий AgentState

Все агенты используют единое состояние.

Во время прохождения графа оно постепенно заполняется.

```
raw_intent
      │
      ▼
intent
      │
      ▼
assumptions
      │
      ▼
env_scenarios
      │
      ▼
attacks
      │
      ▼
threshold_decision
      │
      ▼
bet
      │
      ▼
calibration_record
```

Также состояние содержит

- trust_score
- iteration
- final_decision
- audit_log

что позволяет полностью восстановить историю принятия решений.

# Реализованные агенты

## Intention Agent

Первый агент системы.

Отвечает за преобразование свободного текста пользователя в структурированную бизнес-гипотезу.

Извлекает:

- название идеи;
- описание;
- бюджет;
- горизонт проверки;
- ожидаемый результат;
- бизнес-метрики;
- скрытые допущения.

Связанные таблицы:

```
intents
assumptions
```

При недоступности локальной LLM используется встроенный fallback.

---

## Environment Agent

Анализирует влияние внешней среды.

Формирует возможные сценарии:

- изменение спроса;
- рост стоимости привлечения;
- действия конкурентов;
- изменения законодательства;
- макроэкономические изменения.

Связанная таблица:

```
env_scenarios
```

### Подготовка к RAG

Архитектура агента заранее спроектирована под Retrieval-Augmented Generation.

Используется интерфейс Retriever.

Сейчас:

```
StubEnvironmentRetriever
```

Позже будет заменён на

```
QdrantRetriever
```

без изменения логики самого агента.

---

## Destructor Agent

Получает:

- намерение;
- допущения;
- сценарии среды.

Пытается доказать, что продукт может провалиться.

Для каждого найденного риска строится отдельная атака.

Связанная таблица:

```
attacks
```

---

## Threshold Agent

Является финальным агентом анализа.

Получает результаты всех предыдущих агентов.

Принимает решение:

- Probe
- Hold
- Scale
- Exit

Также рассчитывает:

- обратимость ставки;
- рекомендуемый бюджет;
- уровень доверия;
- обоснование решения.

Связанная таблица:

```
threshold_decisions
```

---

## Calibration Engine

Калибровка запускается отдельно после проведения реального эксперимента.

В отличие от остальных компонентов не использует LLM.

Вычисляет:

- абсолютную ошибку;
- относительную ошибку;
- optimism bias;
- изменение доверия;
- новый trust score.

Связанная таблица:

```
calibration_records
```

---

# Журнал работы агентов

Каждый агент записывает результаты своей работы в audit_log.

Например:

```
Intention

↓

Environment

↓

Destructor

↓

Threshold

↓

Calibration
```

Для каждого шага сохраняются:

- агент;
- тип события;
- полезная нагрузка;
- время выполнения.

Это позволяет полностью воспроизвести весь процесс анализа.


```bash
alembic downgrade -1
```

```bash
alembic downgrade base
```

```bash
alembic current
```
